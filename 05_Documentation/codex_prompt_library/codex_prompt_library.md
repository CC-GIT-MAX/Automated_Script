## 1. 编译错误修复

### 1.1 单文件编译错误

```
当前文件 [文件路径] 在 IAR XX.X 编译时出现以下错误：

[粘贴 build.err 中的错误信息]

请：
1. 定位错误行号
2. 给出修改后的完整代码段
3. 说明修改原因

约束：
- 不要修改函数签名（除非错误是签名本身）
- 不要触碰 [填写不能改的目录]
- 修改后用 git diff 给我看
```

### 1.2 批量编译错误（推荐配合 build.sh）

```
运行 scripts/build.sh，把编译错误修到通过为止。

约束：
- 单次只改 1 个文件，避免连锁错误
- 改一个文件后必须重跑编译验证
- 累计超过 5 次重试仍失败 → 停下来汇报，不要继续乱改
- 不要修改 BSP/ 下的启动文件 startup_*.s
- 不要修改链接脚本 *.icf
- 给我最终的 git diff 汇总
```

### 1.3 链接错误专项

- 中断函数未声明

请先 grep 排查重复符号，告诉我哪些是同一函数在多个文件定义。

## 3. 代码审查

### 3.1 提交前预审

```
按 AGENTS.md 规范审查当前 git diff：

git diff main..HEAD --stat
git diff main..HEAD

重点检查：
1. 命名是否符合规范（小驼峰函数、全大写宏）
2. 是否有未使用变量/函数（IAR Pe177 warning）
3. 错误处理完整性（返回值检查、NULL 指针、数组越界）
4. FreeRTOS 多任务安全（共享资源、临界区）
5. 魔数（时钟分频、延时参数、DAC 参考电压等）
6. 中断函数（不能调用阻塞 API、不能 printf 浮点）
7. 内存使用（栈深度、堆分配）
8. 可重入性

输出格式：按文件分组，每个问题标 [严重/警告/建议]
```

### 3.2 重点模块审查（驱动层）

```
仔细审查 DRV/[模块名]/ 下的所有改动。

检查项：
- 寄存器配置是否与 Reference Manual 章节 [X.Y] 一致
- 中断处理是否清除了所有 pending 位
- DMA 配置：源地址/目标地址/传输长度/模式
- 错误回调是否完整
- 是否有阻塞调用

输出：逐函数分析，每个函数给 1-5 分可信度。
```

### 3.3 性能 / 时序审查

```
审查 [文件路径] 的执行时序：

MCU 频率 [X] MHz，FreeRTOS tick 1ms。

分析：
1. 这个函数最坏情况耗时多少？
2. 是否在中断里调用了它？
3. 是否会被高优先级任务抢占导致超时？
4. 有没有可能进入死循环？

给出优化建议（如果需要）。
```

## 6. Git commit message 生成

### 6.1 Conventional Commits 风格

```
根据以下 git diff 生成 commit message：

git diff --staged

要求：
- 标题 ≤ 50 字符
- 格式：type(scope): subject
- type 范围：feat / fix / refactor / docs / test / chore
- scope 是模块名（如 uart、spi、power）
- 正文说明 what 和 why，不要写 how
- 关联 issue 编号（如果有）
```

### 6.2 中文 commit（团队习惯）

```
根据 git diff 生成中文 commit 信息：

格式：
[类型] 模块名：简要描述

详细说明（可选）

类型：新增 / 修复 / 优化 / 重构 / 文档 / 测试
```

## 8. Bug 定位六问法

当一个 bug 反复出现或难以复现时：

```
用六问法帮我定位这个 bug：

现象：[描述]

1. 第一次出现是什么时候？什么 commit 之后？
2. 触发条件是什么？100% 复现还是偶发？
3. 哪些环境变了（电源、温度、外设型号、固件版本）？
4. 去掉什么条件后 bug 消失？
5. 加什么最小条件后 bug 必现？
6. 同样的代码在哪个模块工作正常、哪个模块不正常？

请基于我的回答给出 3 个最可能的根因和验证方法。
```

## 9. Codex 安全使用红线

### ✅ 可以让 Codex 做

- 改应用层业务逻辑
- 写驱动代码框架
- 生成单元测试
- 修编译错误（除启动文件外）
- 审查代码（预审）
- 生成文档 / 周报
- 分析日志 / 波形

### ⚠️ 必须二审

- 改 BSP 下的驱动实现
- 改 FreeRTOS 配置（config 文件）
- 改链接脚本（*.icf / *.sct）
- 改内存映射
- 优化性能关键代码

### 🚫 禁止让 Codex 直接动

- 启动文件 `startup_*.s`
- 中断向量表
- 时钟初始化（PLL）
- 看门狗 / 复位相关
- Bootloader 烧录流程
- 量产烧录参数
- 客户密钥 / 证书

## 10. 高频组合工作流

### 10.1 编译-修复循环

```bash
#!/bin/bash
# fix_build.sh — 编译错误自动修复循环
MAX_RETRY=5
for i in $(seq 1 $MAX_RETRY); do
  echo "=== 尝试 $i/$MAX_RETRY ==="
  ./scripts/build.sh 2>&1 | tee build.err
  if [ $? -eq 0 ]; then
    echo "✅ 编译通过"
    exit 0
  fi
  # 把错误喂给 Codex
  codex "看 build.err，按 AGENTS.md 规范修复，单次只改 1 个文件"
  if [ $? -ne 0 ]; then
    echo "❌ Codex 修复失败，需要人工介入"
    exit 1
  fi
done
echo "❌ 达到最大重试次数"
exit 1
```

## 11. 提示词调优技巧

1. **具体 > 抽象**：写"修 UART 接收中断里 buffer 溢出"而不是"修 bug"
2. **给上下文**：粘贴相关文件路径、相关 commit、相关 issue
3. **约束要硬**："不要改 startup_*.s" 比 "小心点" 有效
4. **要求输出格式**：要 diff、要列表、要 markdown 表格
5. **分步走**：复杂任务拆成多轮 prompt，每轮只问一件事
6. **要求自验证**："修改后运行 X 验证" 而不是 "修改正确即可"
